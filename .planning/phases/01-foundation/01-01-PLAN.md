# Phase 01 Plan 01

## Objective

Stand up the ForgeSwarm monorepo foundation and a runnable desktop shell so subsequent phases can focus on product workflows instead of infrastructure. This plan should leave the repository with a clean package layout, a Tauri v2 + React 19 + TypeScript + Tailwind desktop app baseline, shared package boundaries, and baseline build, lint, and test commands.

## Planning Assumptions

- Use `pnpm` workspaces as the monorepo package manager.
- Use a Vite-based React frontend inside the Tauri desktop app.
- Keep PTY handling, orchestration, and SQLite integration out of this plan except for placeholder package boundaries.
- Prefer a thin but believable shell over premature feature implementation.

## Task List

1. Create root monorepo tooling and workspace configuration.
   - Add root `package.json` with workspace scripts for `dev`, `build`, `lint`, `test`, and `typecheck`.
   - Add `pnpm-workspace.yaml`.
   - Add shared TypeScript configuration at the root.
   - Add baseline lint and test configuration that can scale across packages.

2. Scaffold the workspace package layout.
   - Create `apps/desktop`.
   - Create `packages/ui`.
   - Create `packages/shared`.
   - Create `packages/agent-runner`.
   - Create `packages/git-orchestrator`.
   - Add minimal `package.json` files and TypeScript entry points for each package.

3. Stand up the desktop shell in `apps/desktop`.
   - Initialize Tauri v2 with a Rust backend.
   - Wire React 19, TypeScript, and Tailwind CSS into the desktop frontend.
   - Add a minimal application shell with dark operational styling that matches the product direction.
   - Add a small Tauri command or health check path to prove frontend-backend wiring.

4. Establish shared application boundaries for later phases.
   - Add shared types for repositories, swarms, agents, tasks, approvals, and checks in `packages/shared`.
   - Add a UI package entry point for reusable components and theme tokens.
   - Add placeholder public interfaces in `packages/agent-runner` and `packages/git-orchestrator` so later phases extend stable contracts rather than refactor from scratch.

5. Add baseline quality and developer workflow support.
   - Ensure the workspace installs and resolves package references cleanly.
   - Add a minimal test harness for shared logic or state utilities where practical.
   - Add root scripts and README starter notes sufficient for local development during phase 2.

6. Verify the phase acceptance criteria before closing the plan.
   - Confirm all required directories exist.
   - Confirm the desktop app can start in development mode.
   - Confirm build, lint, test, and typecheck commands are present and pass or have clearly documented stubs with rationale.

## Verification

- `pnpm install`
- `pnpm build`
- `pnpm lint`
- `pnpm test`
- `pnpm typecheck`
- `pnpm --filter desktop dev` or the equivalent desktop dev command to confirm the shell boots
- Manual check that the repository contains the required monorepo directories and that the desktop shell renders without relying on mock orchestration yet

## Constraints And Deviation Rules

- Do not add live model integration, real orchestration logic, or persistence implementation in this plan.
- Do not introduce unnecessary services or external infrastructure.
- Keep the desktop UI focused on shell-level structure, not completed product screens.
- If a Tauri or Tailwind setup detail conflicts with React 19 compatibility, prefer the simplest stable configuration and document the deviation in the summary.
- If one verification command cannot pass due to upstream template limitations, document the exact failure and keep the rest of the workspace coherent.

## Completion Criteria

- The monorepo structure exists and matches the target package layout.
- The desktop shell is runnable with Tauri v2, React 19, TypeScript, and Tailwind CSS.
- Shared and placeholder package boundaries exist for future orchestration, Git, and UI work.
- Root scripts exist for build, lint, test, and typecheck.
- The repository is ready for phase 2 planning without needing foundational rework.
