# Phase 04 Plan 01

## Objective

Add local persistence and demo data to ForgeSwarm so the repo picker, swarm setup, and workspace shell survive app restarts and can be seeded with a believable showcase session. This phase should move the app from purely in-memory session state to a local SQLite-backed model while preserving the current UX and keeping orchestration, PTY, and merge execution out of scope.

## Planning Assumptions

- SQLite should live behind a narrow storage layer and not leak SQL details into React screens.
- Use local seed/demo data generation to prepopulate repositories, a swarm draft, and one believable workspace session.
- The Rust/Tauri backend should own filesystem-safe DB access and app data path decisions.
- Keep the app browser-verifiable where practical, with a fallback in-memory storage adapter if native DB access is unavailable during frontend verification.

## Task List

1. Define the persistence contract and schema boundary.
   - Add storage-facing shared types or DTOs where needed.
   - Decide how repositories, swarms, agents, tasks, messages, terminal sessions, diffs, approvals, settings, and check runs map into local tables.
   - Keep the schema simple and MVP-oriented, but stable enough for later orchestration integration.

2. Implement the local persistence layer.
   - Add SQLite access in the Tauri backend.
   - Create schema initialization and a minimal migration/bootstrap path.
   - Expose typed backend commands for loading and saving core app state.
   - Keep all writes local-first and synchronous enough for a desktop MVP without introducing unnecessary services.

3. Add a seed/demo data path.
   - Create a deterministic seeded demo repository record, swarm, agents, timeline/check context, and workspace session.
   - Support loading the seed path from the UI so the app can demonstrate the product without any prior state.
   - Make seed generation reuse the shared mock workspace/session factories where possible instead of duplicating fake data.

4. Introduce a frontend storage adapter and app bootstrapping flow.
   - Replace direct in-memory initialization in `App.tsx` with a load/save boundary.
   - Load recent repositories, the latest swarm draft, and the latest workspace session on app start.
   - Persist relevant user actions from phases 2 and 3:
     - selecting/validating repositories
     - swarm draft changes
     - launching a workspace session
     - active agent focus where it improves continuity

5. Integrate persistence into the existing UI flows.
   - Home screen should read recent repositories from persisted state.
   - Swarm Setup should restore the current draft when reopening the app.
   - Workspace should restore the latest saved session and seeded demo session when available.
   - Add an explicit "load demo" or equivalent showcase affordance if it materially improves the MVP path.

6. Add tests and verification around storage and seed behavior.
   - Add unit tests for seed/demo shaping helpers.
   - Add tests for any pure mapping or storage adapter logic.
   - Preserve lint, typecheck, test, and build cleanliness.
   - If native DB verification is blocked in browser mode, clearly separate frontend verification from native persistence verification in the summary.

## Verification

- `corepack pnpm test`
- `corepack pnpm lint`
- `corepack pnpm typecheck`
- `corepack pnpm build`
- Manual verification that:
  - recent repositories survive app reloads in the supported runtime path
  - a swarm draft survives app reloads
  - launching a workspace session persists enough state to reopen the latest session
  - a seeded demo session can be loaded reliably
- If Windows MSVC prerequisites are resolved, rerun `corepack pnpm dev:tauri` or equivalent native verification to confirm SQLite-backed flows end-to-end

## Constraints And Deviation Rules

- Do not implement the orchestration engine in this phase.
- Do not add PTY/xterm.js process management in this phase.
- Do not turn persistence into a generic backend service; keep it local and app-owned.
- If native SQLite access is blocked by the current Windows toolchain issue, preserve a clear adapter seam and document the exact verification gap rather than faking native confirmation.
- Keep the demo seed believable and directly useful for showing the supervised coding workflow.

## Completion Criteria

- Local persistence exists behind a clean storage interface.
- Repositories, swarm draft state, and workspace session state survive app reopen in the supported runtime path.
- A deterministic demo seed session is available.
- The current UX remains coherent while gaining persisted state.
- The repository remains clean under test, lint, typecheck, and build verification.
