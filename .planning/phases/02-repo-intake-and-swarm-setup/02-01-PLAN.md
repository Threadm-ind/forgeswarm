# Phase 02 Plan 01

## Objective

Implement the first real user workflow in ForgeSwarm: selecting a local Git repository, validating it, surfacing basic stack metadata, and configuring a swarm task with agent count, role roster, and execution preferences. This phase should turn the phase 1 shell into a functional Home/Repo Picker and Swarm Setup experience without yet implementing persistence, orchestration, or merge operations.

## Planning Assumptions

- Use mock local data for recent repositories until SQLite arrives in phase 4.
- Prefer a real local folder picker and repository validation path if Tauri plugin support is straightforward; otherwise keep the validation command real and stub the picker only.
- Keep the active workspace, merge queue, and settings views as placeholders while Home and Swarm Setup become functional.
- The missing Windows MSVC toolchain blocks native verification only; frontend and TypeScript work can proceed now.

## Task List

1. Expand shared domain models for repo intake and swarm draft state.
   - Add explicit repository detection results, validation issues, and stack metadata shapes.
   - Add a `SwarmDraft` or equivalent setup model covering prompt, agent count, role assignments, and toggles.
   - Add helper functions for recent repo presentation, stack badge normalization, and role draft defaults.

2. Create a repository validation and stack detection layer.
   - Add a desktop-side service for validating a selected path as a Git repository.
   - Detect basic project signals from common files such as `package.json`, `Cargo.toml`, `pyproject.toml`, `go.mod`, or lockfiles where present.
   - Surface validation states for unsupported, unreadable, missing Git, and valid repositories.
   - Keep the implementation local-first and easy to swap into a persisted flow later.

3. Add repository selection plumbing to the desktop app.
   - Introduce a clear app state container for selected repository, recent repos, validation status, and swarm draft.
   - Integrate a folder-pick action that can later use Tauri dialog APIs without rewriting screen logic.
   - Seed the app with believable mock recent repositories for the MVP while supporting a real selection path.

4. Refactor the app shell into screen-level components.
   - Split `App.tsx` into smaller screen and feature components.
   - Create a functional Home / Repo Picker screen.
   - Create a functional Swarm Setup screen.
   - Keep the Workspace, Merge Queue, and Settings routes as structured placeholders rather than removing them.

5. Implement the Home / Repo Picker screen.
   - Show recent repositories with validation state and detected stack badges.
   - Add an open-folder CTA and a validation result panel.
   - Make failure states readable and operational rather than generic alerts.
   - Route a valid repository into the swarm setup flow.

6. Implement the Swarm Setup screen.
   - Add task prompt input.
   - Add agent count selector limited to 2, 4, 6, and 8.
   - Add editable role roster using the current default role helper as a starting point.
   - Add toggles for auto-run tests, require merge approval, allow file writes, and allow terminal commands.
   - Provide a ready-to-create summary card so the next phase can consume the draft state cleanly.

7. Add practical tests and verification support for the new state flow.
   - Add unit tests for repository validation helpers and swarm draft defaults where practical.
   - Verify the Home and Swarm Setup screens render correctly in browser mode.
   - Preserve existing lint, build, and typecheck cleanliness.

## Constraints And Deviation Rules

- Do not add SQLite persistence in this phase.
- Do not implement orchestration, agent execution, terminal panes, or diff review in this phase.
- Do not hide invalid repository states behind mocks if a real validation path exists.
- If real folder picking is blocked by the current native environment, keep the picker boundary explicit and use a temporary typed input fallback only if it is clearly marked and easy to replace.
- Preserve the dark, dense, operational design direction from phase 1 rather than switching to generic app chrome.

## Completion Criteria

- Home screen supports recent repos, open-folder selection, validation feedback, and stack badges.
- Swarm Setup supports task prompt, agent count, role roster, and required toggles.
- Shared models and helper functions exist for repository intake and swarm draft state.
- The app structure is refactored enough that later phases can add workspace and persistence behavior without another large shell rewrite.
- The repository remains clean under test, lint, typecheck, and build verification.
