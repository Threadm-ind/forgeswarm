## Planning Assumptions

- Merge review should be centered on explicit user supervision, not silent automation.
- Existing workspace diff summaries, checks, comments, and Git isolation records should become the source of truth for the merge queue rather than being duplicated into a parallel data model.
- Merge execution can remain confirmation-gated and minimal, but the app should support approve, reject, and request-changes decisions with audit history.
- Reviewer and tester gate state should remain visible in both the workspace and merge queue so the operator can see why a lane is or is not merge-ready.
- If native Git merge execution cannot be fully verified on this machine, the plan should still produce typed review state, robust UI flows, and well-isolated native command seams.

## Task List

1. Expand the shared review and merge domain model.
   - Add typed records for:
     - merge queue entries
     - review decisions and request-changes outcomes
     - richer approval/audit events
     - diff detail sections or file-level review summaries where needed
   - Ensure workspace state and persisted app state can carry merge-ready, approved, rejected, and changes-requested states cleanly.

2. Add merge-queue shaping helpers and review-state reducers.
   - Derive merge queue entries from isolated agent lanes, diff summaries, checks, and approval state.
   - Add pure helpers for:
     - determining merge readiness
     - marking a lane approved/rejected/requested-changes
     - syncing decision outcomes back into workspace and persisted state
   - Keep this logic testable outside the UI and native shell.

3. Implement supervised approval actions in the desktop app.
   - Add app-level actions for:
     - approve merge
     - reject merge
     - request changes
     - focus a queue item back into the workspace
   - Persist these decisions through the existing app-state boundary.
   - Ensure approval changes update both the workspace review panel and Merge Queue view coherently.

4. Replace the Merge Queue placeholder with a real queue screen.
   - Show pending isolated branches/worktrees with:
     - diff summary
     - check status
     - approval state
     - strategy/branch/worktree metadata
   - Support selecting a queue entry and inspecting enough detail to make the approval decision believable.
   - Keep the visual style dense and operational, consistent with the rest of the operator console.

5. Upgrade the workspace review panel into a real decision surface.
   - Extend the right column so the active agent lane can:
     - display merge readiness
     - show richer diff/check context
     - surface current approval state
     - accept approve/reject/request-changes actions
   - Make reviewer/tester gate failures or pending states clearly explain why merge is blocked.

6. Add native merge-oriented command seams in Tauri where practical.
   - Introduce typed commands for review/merge support such as:
     - retrieving diff detail for an isolated branch/worktree
     - evaluating merge readiness
     - performing a confirmation-gated merge or recording why it was blocked
   - Keep protected-branch safety explicit and reject unsafe operations.
   - If native merge execution is not fully verifiable in this environment, preserve the command contracts and document the exact runtime gap.

7. Add tests and verification around review and queue state.
   - Test merge-queue derivation helpers.
   - Test approval/rejection/request-changes reducers or state helpers.
   - Test any pure diff/check readiness logic.
   - Preserve lint, typecheck, test, and build cleanliness.

## Constraints And Deviation Rules

- Do not weaken protected-branch guarantees.
- Do not introduce a silent or automatic merge path.
- Do not treat the app like a chat client; review actions should stay anchored in structured queue/workspace state.
- If native merge execution remains unverifiable on this machine, keep the domain model, reducers, and UX complete enough that the MVP still feels operational.
- Keep phase scope focused on supervised review and queue behavior; settings and presentation polish still belong to phase 08.

## Completion Criteria

- ForgeSwarm has a real Merge Queue screen backed by current workspace and Git isolation state.
- Reviewer/tester gates, diff summaries, and approval decisions are visible and coherent across the app.
- The operator can approve, reject, or request changes explicitly.
- Merge-readiness logic is typed, testable, and auditable.
- The repository remains clean under test, lint, typecheck, and build verification.
