## Planning Assumptions

- Use typed mock workspace/session data in this phase rather than persistence-backed records.
- Keep the terminal surface UI-ready but do not integrate xterm.js yet unless the package boundary is trivial and does not slow the phase down.
- Preserve the dark, dense control-room look established in phases 1 and 2.
- Agent switching, timeline updates, and touched files can be mock-driven as long as the data contracts are structured for later orchestration integration.

## Task List

1. Expand shared workspace-facing models.
   - Add types for coordinator plan items, timeline events, terminal sessions, touched files, workspace comments, and diff summary/check shells where needed.
   - Add mock factory helpers for a believable seeded workspace state derived from a selected repo and swarm draft.

2. Create a dedicated workspace state layer in the desktop app.
   - Separate workspace/session state from repo selection and swarm setup state.
   - Introduce active agent selection, active terminal selection, and focused review item state.
   - Keep the state transitions simple and explicit so phase 5 can later feed structured orchestration events into the same model.

3. Replace the current placeholder workspace with the real three-column layout.
   - Left column:
     - swarm navigation
     - agent roster
     - task status summary
     - touched files list
   - Center column:
     - active terminal/log surface
     - structured timeline
     - coordinator plan
   - Right column:
     - diff summary shell
     - checks status
     - approvals/comments shell

4. Implement agent switching and contextual detail updates.
   - Clicking an agent should update the active terminal/log content, touched files focus, and right-side review context.
   - Make agent status and role visually clear at a glance.
   - Keep interactions fast and local with no fake loading spinners unless they indicate a real state transition.

5. Improve the route relationship between Swarm Setup and Workspace.
   - Starting a swarm from setup should hydrate the workspace with a believable seeded session.
   - Preserve access to Home and Settings without losing the active workspace shell state.
   - Keep Merge Queue as a lighter placeholder if phase 7 still owns its full implementation, but align its data language with the workspace review panel.

6. Add UI-level tests or state tests where practical.
   - Test seeded workspace-state shaping helpers.
   - Test any non-trivial agent switching or selection logic extracted into pure helpers.
   - Preserve lint, typecheck, test, and build cleanliness.

## Verification

- `corepack pnpm test`

- `corepack pnpm build`
- Manual browser-mode verification that:
  - creating a swarm from setup opens the workspace shell
  - the three-column layout is readable and stable
  - switching agents updates the active session content
  - touched files, timeline, checks, and approval panels remain coherent
- If Windows MSVC prerequisites are resolved, optionally rerun `corepack pnpm dev:tauri` for native verification

## Constraints And Deviation Rules

- Do not add SQLite persistence in this phase.
- Do not implement real orchestration engines, PTY processes, or merge execution in this phase.
- Do not regress the repo picker or swarm setup flow already completed in phase 2.
- If xterm.js integration starts to dominate the scope, keep a high-fidelity terminal placeholder and defer the actual terminal implementation to a later plan.
- Optimize for believable supervision UX, not feature completeness.

## Completion Criteria

- Workspace screen is a real three-column supervision surface, not a placeholder.
- Agent switching updates the active session context coherently.
- Coordinator plan, timeline, touched files, checks, and approval shells are visible and structured.
- The setup-to-workspace transition feels like a believable operator workflow.
- The repository remains clean under test, lint, typecheck, and build verification.
