## Planning Assumptions

- Worktree-first isolation is the preferred MVP path, with branch-only fallback where worktrees are not viable.
- The Git layer should expose typed plans, status records, and confirmation-required actions before it attempts destructive operations.
- Git inspection and setup should be owned by local app code and Tauri commands, not by an external service.
- Phase 6 should prepare believable isolated workspaces and conflict/error surfacing without implementing final merge execution; phase 7 still owns approval and merge queue behavior.
- Native Git verification may be partially blocked by the current machine toolchain gap for Tauri, so the plan should preserve testable pure helpers and browser-safe UI behavior where possible.

## Task List

1. Expand the shared Git and workspace isolation model.
   - Add shared types for:
     - isolation strategy and per-agent workspace state
     - Git setup status, conflict state, and failure surfacing
     - protected branch metadata
     - reset/rollback affordances that require confirmation
   - Extend workspace/session records so the UI can show real isolation state per agent instead of only branch/worktree strings embedded in the agent record.

2. Replace the thin Git helper with a real orchestration-facing isolation module.
   - Refactor `packages/git-orchestrator` from a static planner into a typed service layer that can:
     - derive a safe isolation plan from the selected repository and current swarm
     - choose worktree or branch strategy
     - compute deterministic branch/worktree paths per agent
     - validate protected-branch assumptions
     - surface rollback/reset actions and warnings without executing them yet by default
   - Keep pure planning and normalization logic testable outside the native runtime.

3. Add native Git inspection/setup commands in the Tauri layer.
   - Introduce typed Tauri commands for repository Git inspection and isolation setup.
   - Inspect the selected repository for:
     - current branch
     - worktree support / viability
     - dirty state and basic conflict conditions
   - Implement local setup logic for isolated branches or worktrees where feasible.
   - Ensure failures and unsupported states return explicit typed errors rather than generic command failures.

4. Wire isolation planning and setup into app state.
   - When a workspace launches, derive or request the agent isolation plan for the active swarm.
   - Persist Git isolation metadata alongside the workspace session.
   - Keep the mock orchestration runner compatible with the new isolation model so agent lanes can reflect real branch/worktree status even before PTY execution lands.
   - Avoid mutating the protected branch directly during workspace boot.

5. Upgrade the workspace and merge-adjacent UI for Git visibility.
   - Extend the workspace shell to show:
     - per-agent isolation strategy
     - branch/worktree readiness
     - protected branch status
     - Git warnings, dirty-repo notices, and setup failures
   - Improve Merge Queue placeholder alignment so it reflects pending isolated branches/worktrees and pending approval state using real isolation metadata.
   - Surface rollback/reset actions as visible but confirmation-gated controls if the underlying action is not yet fully executed in this phase.

6. Add tests and verification around isolation logic.
   - Test pure isolation plan derivation in `packages/git-orchestrator`.
   - Test shared normalization/state helpers for agent Git status.
   - Test any reducer or adapter logic that maps native Git results into workspace state.
   - Preserve lint, typecheck, test, and build cleanliness.

## Constraints And Deviation Rules

- Do not implement final merge approval execution in this phase.
- Do not add force-push workflows.
- Do not rely on hidden Git mutations; setup, warnings, and rollback/reset affordances must remain inspectable.
- Do not let Git setup mutate the protected branch directly.
- If full native Git setup cannot be verified on this machine, keep the pure planning/model layers solid and document the exact native verification gap.

## Completion Criteria

- ForgeSwarm has a typed Git isolation model and real per-agent isolation state.
- The app can derive and surface a believable isolated branch/worktree plan per agent.
- Protected branch handling, dirty-state warnings, and conflict/error surfaces are visible to the operator.
- The Git layer is modular enough to support phase 07 diff review and merge approval work without redesign.
- The repository remains clean under test, lint, typecheck, and build verification.
